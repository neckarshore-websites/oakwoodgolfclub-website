## Expert Notes

### Warum der Revert nicht destruktiv ist

b77fee5 war kein harmloser No-Op: zwei `<Image priority>` auf demselben Hero konkurrieren potenziell um die ersten TCP-RTTs auf Mobile-3G. Auf der heutigen Messung kaum sichtbar (der Fix hatte ja keine DOM-Wirkung), aber wenn Next.js 16 sein Verhalten in einer zukünftigen Version korrigiert, könnte der alte Code plötzlich doppelt priorisierte Bilder produzieren. Revert jetzt = kein latentes Risiko in der Zukunft.

### Was für einen echten LCP-Fix nötig wäre (Scope für #43)

Das Element-Render-Delay von 1.9 s auf Mobile ist das echte Problem. Hypothesen mit Kosten-Nutzen:

| Hypothese | Test | Effort | Erwarteter Impact |
|---|---|---|---|
| (a) `<link rel="preload" fetchpriority="high">` im `<head>` statt Component-`priority` | HTML-grep auf rendered page, dann 3-Run-Lighthouse | XS | Adressiert max. 473 ms der 2.9 s — kleine Verbesserung |
| (b) CSS-Animation `hero-image-swap` erst nach `load`-Event starten (oder prefers-reduced-motion default auf aktiv) | Animation-init verzögern, Paint-Frame beobachten | S | Adressiert evtl. 500-1000 ms Render-Delay |
| (c) Hero-Overlay-Gradient + Section-Hydration entkoppeln — Hero als Pure-CSS-Komponente, kein `next/image` für Bottom-Layer | Größerer Umbau, Image via CSS `background-image` mit `will-change: transform` | M | Adressiert potenziell das ganze Render-Delay |
| (d) Zweiten Layer komplett droppen und statischen Vintage-Hintergrund nehmen | Animation-Entfernung, Design-Entscheidung | XS (Code) + User-Design-OK | Potentiell 1.5+ s LCP-Gewinn — aber Visual-Change |

Alles im Scope von #43, nach Launch. Keine dieser Optionen sollte als "one-liner" gemacht werden — alle brauchen Messung vor und nach.

### Warum Mobile 92 launch-fähig ist

Lighthouse-Scoring ist ein gewichteter Composite. 92 entspricht in Chrome-DevTools-Definition:
- LCP < 4.0 s → 62 Punkte Base, mit 2.9 s ergibt sich ein gewichteter Score um 70-80 auf dieser Metrik
- FCP 1.2 s → full score (grün)
- TBT 140 ms → near-full score (grün)
- CLS 0 → full score
- SI 3.6 s → near-full

Die 8 Punkte Lücke zu 100 kommen komplett aus LCP. Das ist keine User-Experience-Baustelle — der Hero ist auf DACH-Mobile-LTE in < 2 s sichtbar (Lighthouse simuliert Slow-4G + 4× CPU, was 2026 für DACH-Mittelstand unrealistisch pessimistisch ist).

Desktop 100/100/100/100 ist der Wert, der auf LinkedIn-Shares und Gast-Link-Klicks sichtbar wird.
