## Expert Notes

### Warum der Runbook nicht in BRIEFING.md oder phase-1-plan.md landete

BRIEFING ist Stage-0-Scope-Dokument, Phase-1-Plan ist Build-Block-Tracker. Beide sind "was-Ebene". Ein Runbook ist "wie-und-wann-genau-Ebene" — anderer Dokumenttyp, andere Lesezeit-Erwartung. Separates File macht es einfacher zitierbar (URL-Anchor auf §2 Prereqs, §5.0 Verify, §6 Rollback direkt per Markdown-Link erreichbar) und separat erweiterbar, ohne die Plan-Docs mit Runbook-Details zu überladen.

### Warum der Verify-Script keinen MX-Check hat

MX-Integrität ist nicht automatisch smoke-bar. Ein MX-Lookup (`dig MX oakwoodgolfclub.de`) sagt nur, welche Hosts aufgeführt sind — nicht, ob eine Mail ankommt. Das einzige verlässliche Smoke ist: User schickt manuelle Mail aus externem Account an `info@oakwoodgolfclub.de`, schaut in IONOS-Postfach. Das ist §5e im Runbook, nicht automatisierbar, deshalb außerhalb des Script-Scopes.

### EXPECTED_APEX_IP bewusst optional

Die Apex-IP von Vercel war lange 76.76.21.21, aber Vercel hat historisch gewechselt (z.B. 216.150.1.1-er-Range in 2024). Ein hardcoded Expected-Wert veraltet den Script still. Mit Optional-Semantik:
- Run ohne Env-Var: Script printed observed IPs, User eyeballt.
- Run mit `EXPECTED_APEX_IP=<from-vercel-dashboard>`: Script assertet. User setzt den Wert **nachdem** sie ihn aus dem Vercel-Dashboard-Widget (Schritt §3b.3) entnommen haben — immer aktuell.
